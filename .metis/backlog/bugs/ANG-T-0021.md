## Bugs Found

### Bug 1 (HIGH): Context manager `__enter__` does NOT install requirements
**File:** `crates/angreal/src/python_bindings/venv.rs:220-224`

`__enter__` calls `create()` and `activate()` but NOT `install_requirements()`. The `venv_required` decorator DOES call it. This means:
```python
# Decorator - requirements ARE installed (works)
@venv_required(".venv", requirements="flask")
def my_task(): ...

# Context manager - requirements NOT installed (BROKEN)
with VirtualEnv(".venv", requirements="flask") as venv:
    import flask  # FAILS!
```

**Fix:** Add `slf.install_requirements()?;` between `create()` and `activate()` in `__enter__`.

### Bug 2 (MEDIUM): `angreal.integrations.venv` not registered in `sys.modules` consistently
**File:** `crates/angreal/src/lib.rs:944-988`

The `angreal` pymodule registers `angreal.integrations.docker` and `angreal.integrations.git` in `sys_modules`, but NOT `angreal.integrations.venv`. The `initialize_python_tasks()` function (line 822) also omits it. Imports work today only because `integrations/mod.rs` does its own registration, but the `initialize_python_tasks()` codepath (used by external tools) will fail.

**Fix:** Add `sys_modules.set_item("angreal.integrations.venv", ...)` in both the `angreal` pymodule function and `initialize_python_tasks()`.

### Bug 3 (MEDIUM): `~` paths are not expanded
**File:** `crates/angreal/src/python_bindings/venv.rs:54-56`

Paths starting with `~` are treated as "absolute" but `PathBuf::from("~/foo")` does NOT expand `~`. `VirtualEnv(path="~/myvenv")` creates a literal `~` directory in the cwd.

**Fix:** Use `shellexpand::tilde()` or Python's `Path.expanduser()` before resolving.

### Bug 4 (MEDIUM): Documentation references non-existent `_create()` method
**File:** `docs/content/reference/python-api/integrations/venv.md:369-370`

Docs show `venv._create()` but the method is `create()` (no underscore prefix).

### Bug 5 (MEDIUM): Documentation incorrectly says activation is "Process-Only"
**File:** `docs/content/reference/python-api/integrations/venv.md:278-279`

Docs claim subprocesses won't inherit activation, but `activate()` modifies `os.environ["PATH"]` and `os.environ["VIRTUAL_ENV"]` which ARE inherited. Test `test_subprocess_uses_venv_python` validates this.

### Bug 6 (LOW): `sys.exec_prefix` not set during activation
**File:** `crates/angreal/src/python_bindings/venv.rs:163`

Activation sets `sys.prefix` but not `sys.exec_prefix`, which some tools require. Standard venv activation sets both.
