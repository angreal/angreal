# Flox Integration: Fix Service Environment Inheritance and Complete Implementation

## Objective

Fix critical service environment inheritance bug, add services support to context manager, and resolve multiple issues preventing the Flox integration from being usable for its primary use case: starting services before running shell commands.

## Bugs Found

### Bug 1 (CRITICAL): Service processes do NOT inherit activated environment
**File:** `crates/angreal/src/integrations/flox/mod.rs:159-175`

`services_start()` spawns `flox services start` via `std::process::Command`, which inherits from Rust's `std::env`, NOT from Python's `os.environ`. The `activate()` method only modifies Python's `os.environ` via PyO3. This means:

1. `activate()` sets env vars in Python-land
2. `services.start()` spawns subprocess from Rust-land with the ORIGINAL environment
3. Python `subprocess.run()` calls DO inherit activated env (from `os.environ`)
4. But Rust-spawned `flox services start` does NOT

The `flox_required` decorator flow (lines 703-757) is:
1. Activate Flox environment → modifies `os.environ` ✓
2. Start services → uses `std::process::Command` which does NOT see `os.environ` ✗
3. Call wrapped function
4. Stop/deactivate

**Fix:** In `activate()` (line 152-153 of flox.rs), also call `std::env::set_var(key, value)` for each variable alongside `environ.set_item(key, value)`. Correspondingly, `deactivate()` must restore `std::env` as well.

### Bug 2 (HIGH): Context manager does NOT support services
**File:** `crates/angreal/src/python_bindings/integrations/flox.rs:195-209`

The design doc (ANG-I-0008) shows this pattern:
```python
with Flox(".", services=["postgres", "redis"]) as flox:
    subprocess.run(["pytest", "tests/integration"])
```

But `Flox.__new__` (line 29) only accepts `path` — NO `services` parameter. `__enter__` only calls `activate()`, not service start. `__exit__` only calls `deactivate()`, not service stop. Users must manually manage services inside the `with` block.

**Fix:** Add optional `services` parameter to `Flox.__new__`. Start in `__enter__`, stop in `__exit__`.

### Bug 3 (HIGH): `services_logs` with `follow=True` blocks indefinitely
**File:** `crates/angreal/src/integrations/flox/mod.rs:259-281`

`services_logs()` calls `cmd.output()` which waits for process exit. With `--follow`, `flox services logs --follow` streams continuously and never exits. Calling `flox.services.logs("postgres", follow=True)` hangs forever.

**Fix:** Either raise an error for `follow=True`, use `cmd.spawn()` for streaming, or remove the parameter.

### Bug 4 (MEDIUM): `~` paths not expanded
**File:** `crates/angreal/src/python_bindings/integrations/flox.rs:47`

Same issue as VirtualEnv — paths starting with `~` treated as absolute but `PathBuf::from("~/foo")` does NOT expand `~`.

### Bug 5 (MEDIUM): `chrono_now()` produces non-ISO timestamp
**File:** `crates/angreal/src/python_bindings/integrations/flox.rs:492-498`

Produces `"1709568000Z"` (bare Unix seconds) instead of ISO 8601 `"2026-01-13T01:30:00Z"` as documented. Doc comment on `FloxServiceHandle` says "ISO timestamp".

### Bug 6 (MEDIUM): `FloxServiceHandle.stop()` stops ALL services
**File:** `crates/angreal/src/python_bindings/integrations/flox.rs:524-532`

`stop()` calls `services_stop()` which runs `flox services stop` without specifying which services. Should only stop the services tracked by the handle.

### Bug 7 (LOW): `ServiceInfo` cannot be instantiated from Python
**File:** `crates/angreal/src/python_bindings/integrations/flox.rs:263-293`

No `#[new]` constructor. Tests have empty `pass` bodies for this class. Limits testability.

## Missing Functionality

1. **No service readiness checking** — After `services.start()`, no wait-for-ready mechanism. Services like PostgreSQL need time to initialize. No `wait_for_ready(timeout=30)`.
2. **No individual service stop** — `services_stop()` in the Rust layer (line 178-193) stops ALL services. The `flox services stop` CLI supports specifying service names but the wrapper doesn't accept them.
3. **No `flox init` or `flox install` wrappers** — Users must shell out manually.

## Implementation Notes

### Technical Approach

**Priority 1 — Fix environment inheritance:**
In `activate()` at `flox.rs:152-153`, add `std::env::set_var(key, value)` alongside `environ.set_item(key, value)`. In `deactivate()`, restore via `std::env::set_var`/`std::env::remove_var`. This ensures `std::process::Command` in the Rust layer inherits the activated environment.

**Priority 2 — Context manager services:**
Add `services: Option<Vec<String>>` to `Flox.__new__`. In `__enter__`, after `activate()`, start specified services. In `__exit__`, stop services before `deactivate()`.

**Priority 3 — Fix `services_logs` follow:**
Change to use `cmd.spawn()` with streaming, or raise `PyValueError` if `follow=True` is passed.

**Priority 4 — Individual service stop:**
Update `services_stop()` in `mod.rs` to accept `&[&str]` and pass service names to `flox services stop`. Update `FloxServiceHandle.stop()` to pass its tracked service names.

**Priority 5 — Service readiness:**
Add `wait_for_ready(timeout)` that polls `services_status()` until services report "Running" or timeout.

### Test Gaps (need new tests)
- Services start/stop/restart/status with actual Flox environment
- `follow=True` log streaming behavior
- Interaction between `activate()` and service process environment inheritance
- Decorator with `services` parameter actually starting services

- Tilde path expansion
