# Fix UV site-packages path discovery on Windows

## Objective

Fix `UvVirtualEnv::site_packages()` to correctly discover the site-packages path on Windows. Currently two Rust unit tests consistently fail on Windows CI.

## Backlog Item Details

### Type
- [x] Bug - Production issue that needs fixing

### Priority
- [ ] P2 - Medium (nice to have)

### Impact Assessment
- **Affected Tests**: `test_site_packages`, `test_install_packages` in `crates/angreal/src/integrations/uv/mod.rs`
- **Reproduction**: Fails on every Windows CI run (all Python versions)
- **Expected vs Actual**: `site_packages()` should return the correct path to the venv's site-packages directory. On Windows, the subprocess call to the venv's Python either fails or returns an incorrect path. The fallback path (`Lib\site-packages`) may not match where UV actually places packages.

## Acceptance Criteria

- [ ] `test_site_packages` passes on Windows CI
- [ ] `test_install_packages` passes on Windows CI
- [ ] Remove `#[cfg_attr(windows, ignore)]` from both tests
- [ ] `site_packages()` returns correct path for UV-created venvs on Windows

## Implementation Notes

### Investigation Needed
- What path does UV actually use for site-packages on Windows?
- Does `Scripts\python.exe` in the venv work correctly in Windows temp directories?
- Does `site.getsitepackages()` return paths matching `sys.prefix` on Windows?
- Is there a casing issue (`Lib` vs `lib`) in the path comparison?

### Affected Code
- `crates/angreal/src/integrations/uv/mod.rs:168` — `site_packages()` method
- `crates/angreal/src/integrations/uv/mod.rs:160` — `python_executable()` method

### Current Workaround
Both tests marked `#[cfg_attr(windows, ignore)]` in commit 23205e5.

## Status Updates

- 2026-03-18: Bug filed. Tests ignored on Windows as workaround. All Linux/OSX tests passing across Python 3.10-3.14.
